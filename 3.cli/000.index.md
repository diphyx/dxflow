dxflow provides a comprehensive command-line interface that adapts to your environment, supporting local execution, remote server management, and automatic instant engine spawning for seamless operations.

::callout{color="blue"}
**Two Execution Types**: Commands run either locally on your machine or remotely on configured servers. Some commands also support automatic instant engine spawning for zero-setup operations.
::

## Local Commands

Commands that always execute on your local machine for configuration and system management.

::callout{color="green"}
**Local Execution**: These commands run locally regardless of any remote configuration, managing your local dxflow installation and settings.
::

### Available Commands

::tabs
  ::tab-item{label="Configuration"}

  - `config get` - Get configuration value
  - `config set` - Set configuration value
  - `config unset` - Remove configuration value

  ::

  ::tab-item{label="Engine Control"}

  - `engine info` - Display local engine information
  - `engine token` - Generate authentication token
  - `engine boot up` - Start local engine process
  - `engine boot down` - Stop local engine process
  - `engine key generate` - Generate RSA key pair

  ::

  ::tab-item{label="Licensing"}

  - `license activate` - Activate license
  - `license create` - Create license
  - `license info` - Display license information

  ::
::

### Common Operations

```bash
# Configuration management
dxflow config set connection-address "https://server.com"
dxflow config get connection-address

# Engine lifecycle
dxflow boot up --daemon
dxflow boot down

# License management
dxflow license info
```

## Remote Commands

Commands that execute on configured remote dxflow servers for distributed infrastructure management.

::callout{color="amber"}
**Remote Execution**: These commands require server configuration and always target the remote server when a connection is configured.
::

### Available Commands

::tabs
  ::tab-item{label="Engine Operations"}

  - `engine ping` - Test connection to remote engine
  - `engine stat` - Display remote engine statistics
  - `engine healthcheck` - Check remote engine health

  ::

  ::tab-item{label="Bridge Management"}

  - `bridge list` - List remote bridge instances
  - `bridge create` - Create remote bridge instance
  - `bridge connect` - Connect to remote bridge
  - `bridge disconnect` - Disconnect from remote bridge
  - `bridge remove` - Remove remote bridge instance
  - `bridge prune` - Clean up all remote bridges

  ::

  ::tab-item{label="Shell Management"}

  - `shell list` - List remote shell instances
  - `shell create` - Create remote shell instance
  - `shell connect` - Connect to remote shell
  - `shell remove` - Remove remote shell instance

  ::

  ::tab-item{label="Proxy Management"}

  - `proxy list` - List remote proxy instances
  - `proxy remove` - Remove remote proxy instance
  - `proxy prune` - Clean up all remote proxies

  ::
::

### Remote Setup

```bash
# Configure connection
dxflow config set connection-address "https://prod.server.com"
dxflow config set connection-key "/secure/prod-key.pem"

# Test connection
dxflow engine ping

# Manage remote resources
dxflow shell create production
dxflow bridge list
```

::callout{color="blue"}
**Advanced Configuration**: See [Remoting](/docs/advanced/remoting) for detailed remote setup, authentication, profiles, and multi-environment management.
::

## Instant Engine Support

Some commands support automatic instant engine spawning for local execution when no persistent engine is running.

::callout
**Smart Execution**: These commands adapt to your environment - remote when configured, local engine when available, or automatic instant engine spawning for local execution only.
::

### How It Works

Instant engine commands follow this execution priority:

1. **Remote Configured?** → Execute on remote server (using server's engine)
2. **Local Engine Running?** → Use local persistent engine
3. **Otherwise** → Spawn temporary instant engine locally

::callout{color="amber"}
**Important**: Instant engines only work for local execution. Remote commands always use the server's running engine, never instant engines.
::

### Commands with Instant Support

::tabs
  ::tab-item{label="Key Management"}

  - `engine key list` - List registered authentication keys
  - `engine key register` - Register authentication key
  - `engine key unregister` - Remove authentication key

  ::

  ::tab-item{label="File Operations"}

  - `object fs list` - List files and directories
  - `object fs create` - Create files or directories
  - `object fs delete` - Delete files or directories
  - `object fs rename` - Rename or move files
  - `object fs upload` - Upload files to storage
  - `object fs download` - Download files from storage
  - `object fs zip` - Compress files into ZIP archives
  - `object fs unzip` - Extract ZIP archives
  - `object fs share` - Share files or directories

  ::

  ::tab-item{label="Workflow Management"}

  - `workflow compose list` - List workflow compositions
  - `workflow compose create` - Create workflow composition
  - `workflow compose remove` - Remove workflow composition
  - `workflow compose start` - Start workflow services
  - `workflow compose stop` - Stop workflow services
  - `workflow compose kill` - Kill workflow containers
  - `workflow compose pause` - Pause workflow containers
  - `workflow compose unpause` - Resume workflow containers
  - `workflow compose containers` - List workflow containers
  - `workflow compose logs` - View workflow logs
  - `workflow compose events` - Monitor workflow events
  - `workflow compose execute` - Execute commands in containers

  ::
::
